# 副作用管理

## 一、副作用定义

**副作用（Side Effect）**是指：组件的核心工作是根据状态渲染 UI（纯计算、无交互），所有超出「渲染 UI」之外的操作，都叫副作用。

> 没有副作用，页面就是静态图片

---

### 常见副作用

- 数据请求：fetch / axios
- DOM 操作：直接修改 DOM
- 订阅：WebSocket / SSE / 事件监听
- 定时器：setInterval / setTimeout / animation
- 存储：localStorage / sessionStorage
- 日志：console.log

---

## 二、副作用对前端的影响

### 1️⃣ 正面影响

- 可以请求课程列表
- 可以提交新增 / 编辑表单
- 可以搜索、分页
- 可以和后端交互

### 2️⃣ 负面影响

#### 内存泄漏

- 组件销毁后定时器/订阅还在运行
- 内存占用越来越多

#### 无限循环

- 依赖项设置错误 → 无限触发 → 崩溃

#### 竞态条件

- 多个请求顺序不确定
- 后发的请求先返回，覆盖正确数据

#### 性能问题

- 频繁触发 → 重复渲染 → 卡顿

---

## 三、需要解决的问题

### 1. 控制执行时机

> 解决：副作用想什么时候执行就什么时候执行

- 只在组件挂载时执行一次
- 只在指定状态（searchParams）变化时执行
- 手动触发执行（增删改后刷新）

### 2. 避免重复执行

> 解决：不做无用功，不浪费性能

- 防止接口无意义重复请求
- 防止函数反复创建（useCallback 解决的问题）

### 3. 防止内存泄漏

> 解决：组件销毁，副作用必须跟着销毁

- 清除定时器
- 取消网络请求
- 移除事件监听

### 4. 保证数据一致性

> 解决：永远使用最新数据，不出现旧数据 bug

- 修复闭包陷阱
- 依赖数组正确管理
- 状态同步更新

### 5. 统一管理副作用

> 解决：代码架构清晰，易维护

- 副作用不写在 UI 组件
- 全部收敛到自定义 Hook（Controller）中
- 逻辑内聚、分层管理

---

## 四、React 官方标准方案

React 专门提供了工具，完美解决以上所有问题：

### 1. 核心工具：useEffect

专门用来执行副作用（请求、定时器、DOM），通过依赖数组精准控制执行时机

```ts
useEffect(() => {
  fetchCourseList(searchParams)
}, [searchParams, refreshKey])
```

### 2. 性能工具：useCallback / useMemo

缓存函数 / 数据，稳定引用，避免副作用无限循环、重复触发

### 3. 安全工具：useEffect 清理函数

```ts
useEffect(() => {
  const timer = setTimeout(() => {}, 300)
  return () => clearTimeout(timer)
}, [])
```

### 4. 架构工具：自定义 Hook（Controller）

- 所有副作用写在 useCourseController
- UI 组件零副作用，纯渲染

## 五、最佳实践

### 核心问题

1. 副作用函数放在哪里
2. 是否需要 return 处理

### 分类维度

| 维度 | 类型 |
|------|------|
| 触发视角 | button 触发 / 响应式触发 |
| 清理需求 | 需要清理 / 不需要清理 |

### 实践原则

> useXXX 全部抽离到 hooks 中，view 只保留"button 类型"副作用
