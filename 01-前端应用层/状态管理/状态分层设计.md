# 状态分层设计（工程级规范）

## 一、最终结论

```text
1️⃣ UI状态 → 组件（view）
2️⃣ 业务状态 → hooks（页面级）
3️⃣ 跨组件/跨页面状态 → store
4️⃣ 瞬时状态 → 普通变量 / ref
```

***

## 二、核心判断标准

```text
这个状态的"作用范围"有多大？
```

***

## 三、四种状态分层

### 🟢 1️⃣ UI 状态（View 层）

**特点：**

- 只影响当前组件
- 纯展示逻辑

**放：** 组件内部 `useState`

**示例：**

```ts
const [open, setOpen] = useState(false)
const [activeTab, setActiveTab] = useState('all')
```

***

### 🔵 2️⃣ 页面业务状态（Hooks 层 ⭐）

**特点：**

- 和业务数据相关
- 可能被多个组件用
- 但只属于当前页面

**放：** 自定义 hooks（useXXX）

**示例：**

```ts
function useCourseList() {
  const [list, setList] = useState([])
  const [page, setPage] = useState(1)
  // fetch + 筛选 + 分页逻辑
}
```

***

### 🟡 3️⃣ 全局状态（Store 层）

**特点：**

- 跨页面共享
- 全局唯一

**放：** zustand / pinia

**示例：**

```ts
user / token / theme
```

**注意：** 列表数据（courseList）❌ 不要放 store

***

### 🔴 4️⃣ 非响应式状态

**特点：**

- 不影响 UI
- 只是临时存值

**放：** ref / 普通变量

**示例：**

```ts
const timer = useRef(null)
let cache = {}
```

***

## 四、判断口诀

```text
纯ui状态→ view
业务状态（需要请求获取数据） → hooks
全局用 → store
```

***

## 五、真实例子

### 登录页

```ts
loading → view
form数据 → view
```

### 课程列表页

```ts
list / page / filter → hooks（useCourseList）⭐
```

### 用户信息

```ts
user / token → store
```

***

## 六、常见错误

### ❌ 错误1：把列表放 store

```ts
courseStore ❌
```

**问题：** 状态污染 + 不必要复杂

### ❌ 错误2：hooks 不抽

```ts
组件里一堆 useState + useEffect ❌
```

### ❌ 错误3：所有 state 都往 hooks 放

**问题：** UI 状态也丢进去 → ❌

***

## 七、标准规范

```text
状态放置规范：

1️⃣ UI 状态（展示控制） → 组件内部
2️⃣ 页面业务状态（列表/分页/筛选） → 自定义 hooks
3️⃣ 全局状态（用户/权限） → store
4️⃣ 非 UI 状态 → ref / 普通变量
```

***

## 八、关键总结

```text
store 不是用来"存数据"的
store 是用来"共享数据"的
```
