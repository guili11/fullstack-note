# 状态分层设计（工程级规范）

## 一、最终结论

```
1️⃣ UI状态 → 组件（view）
2️⃣ 业务状态 → hooks（页面级） controllers
3️⃣ 跨组件/跨页面状态 → store
4️⃣ 瞬时状态 → 普通变量 / ref
```

> **状态归属决定着：**
>
> 1. 初始化位置
> 2. 能够跟业务函数写在一起

***

## 二、核心判断标准

```
这个状态的"作用范围"有多大？
```

***

## 三、四种状态分层

### 🟢 1️⃣ UI 状态（View 层）

**特点：**

- 只影响当前组件
- 纯展示逻辑
- **它只负责：画页面、绑定事件、渲染数据**

**放：** 组件内部 `useState`

**示例：**

```ts
const [open, setOpen] = useState(false)
const [activeTab, setActiveTab] = useState('all')
```

***

### 🔵 2️⃣ 页面业务状态（Hooks 层 ⭐）

**特点：**

- 和业务数据相关
- **控制核心业务状态**

**放：** 自定义 hooks（useXXX）

**示例：**

```ts
function useCourseList() {
  const [list, setList] = useState([])
  const [page, setPage] = useState(1)
  // fetch + 筛选 + 分页逻辑
}
```

***

### 🟡 3️⃣ 全局状态（Store 层）

**特点：**

- 跨页面共享
- 全局唯一

**放：** zustand / pinia

**示例：**

```ts
user / token / theme
```

> **注意：** 列表数据（courseList）❌ 不要放 store太大了

***

### 🔴 4️⃣ 非响应式状态

**特点：**

- 不影响 UI
- 只是临时存值

**放：** ref / 普通变量

**示例：**

```ts
const timer = useRef(null)
let cache = {}
```

***

## 四、判断口诀

```
纯ui状态 → view
业务状态（需要请求获取数据，影响请求（表单，id等）） → hooks
全局用 → store
```

***

## 五、状态管理

### 响应式 vs 非响应式

**判断是否需要响应式，就两点：**

1. 是否影响 UI 重新渲染
2. 是否需要触发 useEffect / watch 等副作用钩子重新执行

> 绝大部分是需要响应式的，比如列表数据、筛选条件、分页信息等
> 但是这里，这个 course 就跟一个函数参数一样，不需要响应式

**示例：**

```ts
// 假如有课程，这里就是数据源（不需要响应式）
let course: Course | null = null
```

***

### 状态更新策略

- **immutable**：不改变原数据，返回新数据
- **mutable**：改变原数据

> **这里注意：** 响应式的更新是 immutable 的，才能触发响应式，也就是说 React 是浅监听

**示例：**

```ts
// 响应式更新 - immutable
const [searchParams, setSearchParams] = useState<CourseQueryParams>({
  keyword: '',
  pageSize: 5
})

//react的响应式更新一定是给一个新对象（object.xxx更新没用）

// AntD Form - mutable
// form是 antForm内部数据，不是响应式的，是一个mutable的对象实例
const [form] = Form.useForm<Course>()
```
