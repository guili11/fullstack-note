# 并发

## 为什么需要并发
当任务可以并行执行时（如同时请求多个接口、复杂计算可拆解），并发能显著减少总耗时，充分利用多核 CPU 资源，提高程序吞吐量 。


Go 的实现：协程（Goroutine）

## 核心问题

### 如何给协程分配资源？如何从协程中获取结果？

- 传参数
- 通道（Channel）
- 全局共享内存（Go 不推荐）

### 如何控制协程的生命周期？

**2 点基础理论：**

- 父协程控制子协程的生命周期，父死，子死

### 如何决定多个协程之间的执行顺序？

开始时间好决定，关键就是关闭时间不好决定（使用阻塞控制协程的关闭）

**思想：**

- 父协程阻塞等待子协程执行完毕
- 后置协程阻塞等待前置协程执行完毕，再执行（例如生产端和消费端）

## 协程协作模式

### 父子协程

父协程开启子协程，天然先于子协程执行，但是有一个问题是：父协程不会等待子协程执行完毕。

**解决方案：** 使用 `sync.WaitGroup`

```go
wg.Wait()   // 等待所有子协程完成
wg.Add(1)   // 增加计数
wg.Done()   // 减少计数
```

### 兄弟协程

看业务顺序，一般使用单向 channel 来控制：

```go
func (ch chan<- int)  // 前置协程（只发送）
func (ch chan int)    // 后置协程（只接收）
```

后置协程中有 `value <- ch` 的存在，直到前置协程中 `ch <- value` 执行后，才会解除阻塞。

## 通道关闭时机

由于 `value <- ch` 会阻塞代码，同样 `ch <- value` 也会阻塞代码，所以需要考虑何时关闭通道。

### 最佳实践

实际就是：**发送-接收一致，然后再关闭通道**（实际都不需要手动关闭了）

- 发送不会空发
- 接收不会阻塞

### 特殊情况

#### 1. 无限收集器

接收方：用 `range` 监听通道

```go
func receiver(ch <-chan int) {
    // 重点：range 会一直等！不关闭通道就死循环阻塞
    for val := range ch {
        println(val)
    }
    println("通道已关闭，循环退出") // 只有关闭才会走到这里
}
```

#### 2. 多协程监听一个通道

假如我想发送一个 stop 信号，那不知道发几个，对吧？

我通过关闭通道，这个作为一个信号，所有监听通道的协程都会收到这个信号，然后退出循环：

```go
func watcher(ch <-chan int) {
    // 重点：range 会一直等！不关闭通道就死循环阻塞
    for val := range ch { // 不读任何值
    }
    println("通道已关闭，循环退出") // 只有关闭才会走到这里
    // 执行后续的关闭服务的逻辑 例如 defer 等
}
```

### 手动关闭通道

特殊情况下，无法确保发送-接收一致，需要手动关闭通道。

**手动关闭通道产生的问题只有一个：空发的问题**

**解决方案：** 只需要保证关闭后不再发送数据即可：

> **只能由【发送方】关闭通道**，接收方绝对不能关  
> 不能重复关闭，不能向已关闭的通道发数据（直接 panic）

## 生产消费模型

### 1. heavy，light 与交叉行为

`m:1:(n:1)`（m，n 代表着多协程）

#### 问题抛出：什么时候需要开启多协程呢，也就是并发呢？

- **a 例如**：我要同时请求十个接口的数据
- **b 例如**：我要执行一个超级无敌复杂的计算任务（就比如 HTML 服务端拼接吧）

**如果不并发：**

- a 要耗费 10 个请求加起来的时间
- b 要耗费 1 本身复杂计算的时间

#### 使用并发的一个一贯做派：拆解任务，开启多协程

- a 就是 10 个独立的任务，无需拆解，直接开启多协程
- b 是一个复杂的多计算，可以拆解，然后再开启多协程

#### 场景示例

任务 a + 任务 b + 任务 c（任务 1）  
总共有十个任务 1，最终需要汇总任务结果

**并发方案对比：**

1. 先开启十个任务 a 协程，然后再开启十个任务 b 协程，再开启十个任务 c 协程，最后再汇总到一个协程中
2. 直接开启十个任务 1 协程，然后汇总到一个协程中

**分析：**

- 由于 1 里边 b c 协程都需要等待前置协程，实际上就等同于 2 里边直接一个协程，1 还多了协程开销性能问题
- 所以说我们要注意任务合并（一个任务只开一个协程，无论任务多大（除非本身特别复杂又要开启并发哦），只有最后汇总的时候才使用 chan，阻塞后置汇总协程）  
  这里就是所谓的 `n+1` 了

#### `m+1+(n+1)` 的由来

实际就是：任务汇总之后，后续还要执行一个复杂的大任务，这里需要拆分，然后并发解决。然后呢，这里的拆分都是那个消费协程内部的东西了。

### 总结

无论是生产消费模型，还是单任务拆解，统统都是 `n:1` 模型（实际单任务拆解也可以划分为生产消费模型哦）。

### 2. 生产消费端的速度问题

### 3. chan 的特性

默认下一个 chan 容量就是 1，这个 chan 假如有一个发送，必须有一个接收，才能接着发送。

#### 假如生产消费速度一致

这样自然最好。

#### 假如速度不一致

**消费快**：这个没什么，本身后置任务就是需要等待前置任务，阻塞是正常的。

**生产快**：这个呢，就是生产太快，消费不过来，导致发送数据行为阻塞了生产端后续的代码。

**解决方法：** 增加通道容量，例如：

```go
ch := make(chan int, 100)  // 只有当通道容量满了，才会阻塞生产端的代码
```

#### 生产端阻塞示例

```go
// 生产端协程：仅演示无缓冲通道的阻塞，影响自身后续行为
func producer(ch chan int) {
    // 第一步：快速生产3个数据
    for i := 1; i <= 3; i++ {
        println("生产数据：", i)
        ch <- i // 🔴 无缓冲通道：发送即阻塞！消费端不取走，这里永远卡着
    }

    // ======================================
    // 🔴 关键：以下是【生产端的后续行为】
    // 因为无缓冲通道阻塞，这段代码 永远执行不到！
    // ======================================
    println("【生产端】所有数据生产完成！")
    println("【生产端】开始执行后续业务逻辑...")
    println("【生产端】关闭资源、记录日志...")
}
```
