# Go 协程核心知识体系（按你的三类划分）
## 1. 协程（Goroutine）
- 本质：Go 运行时调度的**轻量级执行单元**，基于 GMP 模型，非操作系统线程，创建销毁成本极低、栈动态伸缩。
- 启动方式：`go 函数()`，异步执行，不阻塞当前协程。
- 核心特性：主协程退出，所有子协程会直接终止；单个协程 panic 不处理会导致整个进程崩溃。
- 配套保障：`panic/recover` 用于协程内独立捕获异常，避免进程退出。
- 常见问题：**协程泄漏**（协程永久阻塞在 channel/锁上，无法退出）。

---

## 2. 协程共享机制（锁 + Channel）  --生产消费者模型
核心目标：**多个协程安全访问/传递共享数据**，解决数据竞争问题。
### （1）锁（共享内存通信）
- 适用场景：多个协程直接操作同一块内存数据，无数据传递需求。
- `sync.Mutex`：互斥锁，完全独占，读写均互斥，保证临界区安全。
- `sync.RWMutex`：读写锁，读共享、写独占，适合读多写少场景，性能更优。
- `sync/atomic`：原子操作，轻量级无锁方案，针对基础数值类型的简单并发修改。
- `sync.Once`：基于锁封装，保证共享初始化逻辑**只执行一次**。

### （2）Channel（用通信共享内存）
- 适用场景：协程之间传递数据，天然无竞争，Go 推荐的并发方案。
- 无缓冲 channel：同步阻塞，收发必须同时就绪。
- 有缓冲 channel：异步队列，缓冲满发送阻塞、缓冲空接收阻塞。
- 基础操作：发送`ch<-`、接收`<-ch`、关闭`close(ch)`；禁止向已关闭 channel 发送、重复关闭。
- `select`：多路复用多个 channel，实现非阻塞读写、多 channel 调度。

## 3. 协程协调机制（WaitGroup + Channel）
核心目标：**调控协程的执行顺序、生命周期、同步通知**，不关注数据共享，只关注“谁先谁后、谁等谁”。
### （1）sync.WaitGroup
- 作用：等待一组协程全部执行完毕，解决主协程提前退出、协程执行顺序管控问题。
- 核心方法：`Add(n)` 登记协程数、`Done()` 协程完成标记、`Wait()` 阻塞等待所有协程结束。
- 定位：纯协程生命周期协调，不涉及数据传递。

### （2）Channel（协调专用）
- 通知协调：用空 channel（`chan struct{}`）实现协程启动、停止、唤醒的信号通知。
- 顺序协调：通过 channel 阻塞特性，强制协程执行先后顺序。
- 超时协调：结合 `time.After` 实现协程执行超时控制。
- 批量协调：作为信号队列，控制协程并发数量（worker 池限流）。

### （3）延伸封装
- `context`：基于 channel 实现的嵌套协调工具，支持协程树的统一取消、超时。
- `errgroup`：WaitGroup + context 组合，实现一组协程的错误协调，一个报错全部取消。


### 4. 并发数据通信场景：生产者 - 消费者模型
核心结论：所有协程间传递业务数据的场景，本质只有生产者 - 消费者模型，无第二种
（1）核心设计原则
单向职责划分
严格区分生产者（仅生产 / 发送数据）、消费者（仅消费 / 处理数据），职责不交叉，避免逻辑混乱
按任务 heavy 程度分配协程数量
根据生产 / 消费逻辑的计算量、IO 阻塞程度，决定开启协程数量，匹配处理效率
注意：只有当需要数据交叉时，才会来到消费端，否则一切操作都在生产端（然后再根据heavy程度分配协程数量）
（2）经典任务分配模式
n+1 模式
n 个生产者协程 + 1 个消费者协程，适用于多源数据生产、单节点汇总处理的场景
n+1+m+1 模式
n 个生产协程 → 1 个中转分发协程 → m 个消费协程 → 1 个结果汇总协程，适用于分片并发处理、多级流转、最终聚合的重型业务场景
 
 
### 5、不是“数据通信”，但属于「并发协作」的模型
 
这些是协程之间互相“控制行为”，不传业务数据，所以不算你理解的“通信”：
 
1. 互斥模型（Mutex）   （多读少写时作为chanel的替代）
 
- 不是传数据，是抢同一资源的使用权
- 比如：修改同一个全局变量、写同一个文件
- 核心：排他，不通信
 
2. 同步等待模型（WaitGroup/信号量）
 
- 协程之间只做一件事：等你做完我再动
- 不传数据，只同步执行节奏
 
3. 广播/通知模型（Cond / 事件通知）
 
- 一个协程喊：“条件满足了，你们可以跑了”
- 只发信号，不传业务数据
- 比如：配置更新后通知所有协程
 
4. 发布-订阅模型（Pub/Sub）
 
- 本质是生产者-消费者的一对多扩展
- 一个生产者广播，多个消费者订阅
- 依然没跳出生产消费的框架
 
5. 原子/无锁竞争
 
- 完全不通信、不等待，直接靠硬件保证修改安全
- 比如计数、开关状态

6. 通道信号协作模型（Channel 专用）【你补充的核心】
作用：纯 Channel 控制协程行为，无任何业务数据传输
实现：chan struct{} 空通道（零内存开销）
场景：协程启动 / 停止信号、执行卡点、超时控制、并发限流
核心：信号同步、行为阻塞 / 唤醒，是 Go 最常用的协作方式

 
 
 
三、给你一句最本质的总结（完全贴合你的理解）
 
1. 只要是「传递数据、处理数据、汇总数据」
→ 全世界只有生产者-消费者模型，没有例外。
2. 只要是「协程之间等对方、抢资源、发通知」
→ 才是互斥、同步、广播这类控制模型。
 
你之前做的n+1+m+1 分片、汇总、并发处理，从头到尾都只属于生产者-消费者，其他模型只是用来辅助控制它的。
