前边已经说过了，依靠路由管理器，管理路由跟handler匹配，并且接收和发送请求，
接下来深入一个完整的服务流程
例如一个服务，进入时需要登录校验，运行时需要日志记录，需要解决跨域问题。
这三个需要的能力其实是绝大多数服务都需要的，而且不同服务，这些能力的代码也都是一样的。

这里其实就是一个最基础的，能力剥离，这也是框架的设计原则，重复的逻辑 剥离出来，用搭积木的方式使用。 

实际应用场景
1. 用户认证：在认证中间件中验证用户身份，然后将用户ID和角色设置到上下文中，供后续处理函数使用。
2. 日志记录：在请求开始时记录时间戳，在请求结束时计算处理时间并记录日志。
3. 错误处理：在中间件中捕获处理函数中的错误，并统一处理。
4. 权限控制：基于用户角色进行访问控制。 


接下来就两点，如何写中间件，以及如何使用中间件

## 中间件怎么写：
中间件本质是一个函数，函数的参数是一个gin.Context，返回值是一个gin.Context。
中间件的作用是在请求处理过程中，对请求进行处理，或者对响应进行处理。

需要注意的核心问题--中间件的执行时机
核心结论：两个函数+一个注册顺序
1. 注册顺序 → 控制谁先执行、谁后执行
2. c.Next() → 控制请求前 / 请求后执行
3. c.Abort() → 控制直接中断，不往下执行 ，同时返回一个错误码
- 执行顺序由调用方决定，一般有一个一般顺序：日志记录 → 跨域处理 → 错误处理 → 认证校验 → 权限校验 → 业务处理
- 中间件实现内部只需要判断根据实际情况，是否放行（登录认证ok），或者是否中断（不ok）

一、规则 1：注册顺序 = 执行顺序（谁先写，谁先跑）
中间件的执行顺序，完全按照你挂载的先后顺序来！
全局中间件 → 分组中间件 → 单个路由中间件 → 业务接口
同一层级：先 Use 的先执行
```go
r.Use(LoggerMiddleware())
// 先登录校验，再管理员校验
admin.Use(Auth(),Admin())
//单个路由中间件最后执行
admin.GET("/user",RateLimit(),getUserList)
```
二、规则 2：c.Next () 控制「请求前 / 请求后」

三、规则 3：c.Abort () 控制「直接中断，不往下走」

# 中间件的挂载方法
只有三种：
· 全局：r.Use(中间件)
· 分组：分组.Use(中间件)
· 单个路由：路由.方法(路径, 中间件, 处理函数)
- 前边两个Use的本质其实也是路由.方法(路径, 中间件, 处理函数)，只是这里用简便的写法，没有写路径。
