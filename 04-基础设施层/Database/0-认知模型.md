## 一、数据库本质

1. 数据库和普通文件（JSON / TXT）的本质区别是什么？

**预先说明：数据库的核心是，在多用户环境下，高效且可靠地管理数据的读写**

跟数据库有关的一切东西，都是为了实现这个目标。
总结几个特点：并发安全，查询性能强大，数据一致性。

本质区别其实就是对于数据来思考
增删改查：
文件：需要自己写查找逻辑
数据库：直接写SQL查找，更加方便

- 本质：数据库本身提供了查找能力，而文件只是存储载体

性能，主要就是查找性能嘛：
文件：通常是全量扫描（O(n)）
数据库：有索引（接近 O(log n)）

- 关键：数据库有索引，能在“大数据量”下仍然高效查询
  并发：
  同时操作同一个数据的情况，如何保证并发安全呢
  文件：基本没有
  数据库：锁/mvcc/事务
  数据一致性：
  文件：写一半崩了 → 数据直接坏掉
  数据库：有事务（要么全成功，要么全失败）

总结
数据库=存储+查询性能强+并发控制+数据一致性

1. 数据库最核心解决的问题是什么？（不仅仅是“存数据”）
   总结为：在多用户环境下，高效可靠地管理数据的读写。
   细致划分其实就是上面的那些特点。
   多用户：并发
   高效：提供查询能力和性能优化（索引，缓存等）
   可靠：保证数据的一致性（事务）

数据库本质上解决的是数据管理问题——在多用户并发场景下，既要保证数据访问的效率，又要保证数据的一致性和可靠性。

## 二、整体架构

作为一个复杂的应用，MySQL是分多层的。类似前端的三层view controller service

1. MySQL 分为哪几层？每一层的职责是什么？

```
1. SQL进来 → 解析语句？
2. 具体怎么执行？
3. 谁去真正拿数据？
```

### MySQL分层（类比理解）

可以类比为一个公司：

1. SQL处理层（策划部门）
   负责解析SQL，并生成执行计划（选择最优执行路径）
   本质是：在多种方案中选择成本最低的执行方式
2. 执行层（项目经理）
   按照执行计划，调度存储引擎完成数据读取
3. 存储引擎层（工厂）
   负责数据的实际存储和读取（包括索引结构实现）

- 当数据库出现问题的时候从这三层里去查找问题

1. 为什么 MySQL 要设计“存储引擎”这一层？

我的理解：这一层次划分出来可以方便替换

存储引擎不仅是执行层与底层存储之间的交互层，
更核心的作用是：让数据的存储方式和行为可以**灵活替换**。

不同业务场景，对数据库的需求不同，例如：

- 有的需要强事务支持（如电商）
- 有的更关注读写性能（如日志系统）

因此 MySQL 通过“存储引擎”这一层，将：

- 数据存储方式
- 索引结构
- 事务机制

进行封装，使执行层只依赖统一接口，而不关心具体实现。

要意识到一个设计思想：

> 👉 MySQL 是“框架”，存储引擎是“插件”
> 这就是典型的：

- 解耦
- 可扩展架构
  假如没有拆分出来这一层次，那么我的这个应用就是死的，只能适配一种应用场景

### 一句话总结

存储引擎 = 可替换的数据存储与管理实现

## 三、SQL执行流程

1. 一条 SQL 从输入到返回结果，完整执行流程是什么？

```
1. SQL解析
SQL语句进入MySQL后，首先进行语法解析（Parser），将SQL转换为内部结构

2. 语义校验
检查库、表、字段是否存在，以及权限是否合法

3. 优化器生成执行计划
优化器基于成本模型，选择最优执行方案（如索引选择、join顺序等）

4. 执行层执行
执行层根据执行计划，逐步调用存储引擎
<!-- 逐步这个说法原因是，数据是一行一行拿到的，也就是计划一个一个执行 ，一步一步调用存储引擎的，  而不是一次性 拿到所有数据   这中间还有数据排序啥的  -->
5. 存储引擎处理
存储引擎负责实际的数据读取（如通过索引或全表扫描）

6. 返回结果
将查询结果逐步返回给客户端
```

1. 优化器在这个过程中解决了什么问题？如果没有它会怎样？

一条sql有多种可选的执行路径， 其实就是最短路径问题嘛 （最低成本）

优化器作用案例：

SELECT \* FROM user u JOIN order o ON u.id = o.user\_id;
先查 user 再 join order
先查 order 再 join user
用哪个索引
用什么 join 算法

优化器会基于成本模型（IO、CPU等）评估不同方案，并选择最优路径。

记住：

> SQl是声明式语言，不是过程式语言

没有了优化器，那我们得自己写具体选择了，多麻烦啊。

## 四、数据存储

1. 数据在磁盘上是如何组织的？（行 / 页 / 文件）

数据库中的数据不是按“行”直接存储的，而是分层组织：

文件（表空间） → 页（Page） → 行（Row）

- 表数据存储在文件中
- 文件被划分为固定大小的页（Page）
- 每一页中包含多行数据

### 为什么按页存储？

因为磁盘IO是按块进行的，无法只读取一行数据，
数据库通过“页”作为最小单位进行读写，以提高效率。

假如一行一行读取，这带来的是高频词的io，也就是频繁的访问磁盘，狗屎一样，占用高

### 一句话总结
数据库以“页”为单位管理数据，以适配磁盘IO特性

2. 为什么数据库要“按页（page）读写”，而不是按行？
数据库按页读写的根本原因是磁盘IO的特性。

1. 磁盘IO的最小单位是“块”，无法按单行读取
即使只查询一行数据，也必须读取一整块数据

2. 顺序读远快于随机读
如果按行读取，会产生大量随机IO，性能极差
而按页读取可以减少IO次数，提高效率

因此数据库选择“页”作为最小读写单位，以适配磁盘特性

提高：
> 所有数据库性能问题，本质都和“页”有关
为什么全表扫描慢？ → 页太多
为什么索引快？ → 少读页
为什么回表慢？ → 多读页
