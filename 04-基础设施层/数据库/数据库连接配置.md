
# 数据库连接配置

数据库操作的两大核心伙伴：驱动层 + 操作层
1. 数据库驱动库（负责：怎么连上、用什么协议）
对应：go-sql-driver/mysql、postgres、sqlite 等
干的事：
实现对应数据库的底层通信协议
识别 DSN 里的地址、账号、密码
建立真实网络连接
一句话：驱动管「物理连接」和「协议翻译」
2. 数据库操作库（负责：连接池管理 + 执行 SQL）
原生：database/sql（*sql.DB）
进阶：gorm（基于前者封装的 ORM）
干的事：
管理连接池（开多少连接、空闲多久、多久回收）
提供统一的 SQL 执行接口，只具备执行能力（语句得根据不同的数据库类型，自己写，这也是为什么使用gorm的核心之一）
GORM 额外做：结构体映射、自动拼 SQL、链式调用
一句话：操作层管「连接调度」和「SQL 执行规范」

简而言之

驱动：告诉程序 → 「我要连这个 MySQL 数据库，用这个账号密码」
操作库：告诉程序 → 「我帮你管好这些连接，不让它乱掉」

DSN、连接池参数、超时设置   

dsn：database source name 驱动配置
连接池参数，超时设置： 操作配置，影响连接池的运行效率

示例代码：
核心，通过sql.Open获取到db对象（*sql.DB）

- 实际代码中，db对象（*sql.DB）是超级管家，是配置的核心对象，能够帮助驱动配置
	// sql.Open 是操作库函数，仅加载驱动配置
	db, err = sql.Open("mysql", dsn)

```go
package main

import (
	"context"
	"database/sql"
	"fmt"
	"log"
	"time"

	// 驱动库（必须导入）
	_ "github.com/go-sql-driver/mysql"
)

// 全局数据库连接池（*sql.DB 是操作库的核心对象）
var db *sql.DB

// InitDB 数据库初始化：拆分 驱动配置 + 操作配置
func InitDB() {
	// ===================== 1. 驱动配置（DSN）：决定【连哪里、怎么连】=====================
	// 属于驱动库的配置：地址、账号、密码、字符集、时区、时间解析
	dsn := "root:你的数据库密码@tcp(127.0.0.1:3306)/test_db?charset=utf8mb4&parseTime=True&loc=Local"

	// ===================== 2. 初始化连接池（操作库底层对象）=====================
	var err error
	// sql.Open 是操作库函数，仅加载配置，不建立真实连接也无法测试连接
	db, err = sql.Open("mysql", dsn)
	if err != nil {
		log.Fatalf("驱动配置加载失败: %v", err)
	}

	// ===================== 3. 操作配置（连接池/超时）：决定【连接效率、稳定性】=====================
	// 以下参数直接影响连接池运行效率、资源占用、并发能力
	db.SetMaxOpenConns(20)                  // 最大打开连接数（并发核心）
	db.SetMaxIdleConns(10)                  // 最大空闲连接数（复用连接，提升性能）
	db.SetConnMaxLifetime(1 * time.Hour)    // 连接最大存活时间（防止旧连接失效）
	db.SetConnMaxIdleTime(30 * time.Minute) // 空闲连接最大超时时间（释放闲置资源）

	// 超时配置：带上下文的Ping，5秒超时（操作层面的超时控制）
	ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
	defer cancel()

	// ===================== 4. 验证连接 =====================
	err = db.PingContext(ctx)
	if err != nil {
		log.Fatalf("数据库连接失败: %v", err)
	}

	fmt.Println("✅ 数据库连接初始化成功")
	fmt.Println("├── 驱动配置(DSN)：完成")
	fmt.Println("└── 操作配置(连接池/超时)：完成")
}

func main() {
	// 初始化
	InitDB()
	// 程序退出时关闭连接池
	defer func() {
		_ = db.Close()
		fmt.Println("✅ 连接池已关闭")
	}()

	// 后续业务CURD...
}
```



# gorm配置
gorm是基于sql.DB的封装，但是只是简化了sql操作，对于原生就很完善的  连接池配置等，是没有额外的新代码的，所以在实际配置的时候要先拿到原生db对象（*sql.DB），再进行连接配置

分工，sql.DB 管底层连接 ， gorm.DB 管上层操作  （add操作...）

## sql.DB与gorm.DB的区别：
GORM 管上层操作，sql.DB 管底层连接
想管连接 → 必须取出 sqlDB

对比维度	sql.DB (底层原生)	gorm.DB (ORM 封装)
归属来源	Go 官方标准库 database/sql	第三方 ORM 框架 gorm.io/gorm
核心定位	数据库连接池本体真正的底层连接管理者	ORM 操作工具上层语法封装、SQL 生成器
核心职责	1. 管理所有数据库连接2. 维护连接池参数3. 建立 / 关闭网络连接4. 驱动通信、执行原生 SQL	1. 结构体与数据表映射2. 自动生成 SQL 语句3. 链式 CURD、关联查询4. 简化业务开发
关键方法	SetMaxOpenConnsPingCloseQuery/Exec	First/Create/Update/DeleteModel/Where/Joins无连接池相关方法
连接能力	具备真实网络连接能力	无连接能力，完全依赖 sql.DB
配置作用	配置连接池、超时、连接生命周期	配置日志、事务、预编译、命名策略
依赖关系	被 gorm.DB 依赖，是底层根基	基于 sql.DB 封装，脱离它无法运行
使用方式	全局单例，程序启动初始化	全局单例，基于 sql.DB 生成

## GORM 核心配置：gorm.Config，
这里参数有sql.DB但是实际上只是建立联系的桥梁，实际上人家的配置是基于sql.DB的，

db=gorm.Open(mysql.Open(dsn), &gorm.Config{})

db=gorm.Open(mysql.Open(dsn), &gorm.Config{})
1. 分工：连接靠底层驱动，ORM 行为靠 gorm.Config；
2. mysql.Open(dsn) 替代原生 sql.Open，为 GORM 指明数据库类型；
3. 关键：sql.DB 只管连接，不认数据库语法；GORM 需自动生成 SQL，必须显式指定数据库类型。


```go
// GORM 把原生的 2 个参数，包装成了 2 个更高级的对象.底层实际就是sql.Open()
db=gorm.Open(mysql.Open(dsn), &gorm.Config{
  SkipDefaultTransaction: true,
  FullSaveAssociations:   true,
})  

// 关键：从gorm.DB中取出sql.DB
sqldb=db.DB()
```


核心概念差异 ：
sql.DB mysql.DB  gorm.DB

问题：
既然sql库 提供统一的 SQL 执行接口呢？那为啥还这么说呢 sql.DB 只管连接，不认数据库语法；GORM 需自动生成 SQL，必须显式指定数据库类型。


database/sql 的统一接口 = 统一「怎么执行 SQL」sql.DB 不认语法 = 不关心「SQL 内容写的是什么」GORM 需要数据库类型 = 必须知道「怎么自动生成 SQL」
