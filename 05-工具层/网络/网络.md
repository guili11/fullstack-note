一、HTTP/HTTPS 基础
HTTP 协议
  ○ 请求方法：GET / POST / PUT / DELETE / PATCH
  ○ 状态码：200/301/302/304/400/401/403/404/500
  ○ 请求/响应头：Content-Type, Cache-Control, Set-Cookie, Authorization,token
  ○ HTTP 特性：无状态、明文传输、请求—响应模式  （一个请求一个响应）
HTTPS / TLS   （http  使用 tls）
  ○ 为什么安全（加密 + 完整性 + 身份验证）
  ○ TLS 握手流程（证书、对称/非对称加密、密钥交换）
  ○ 浏览器安全策略：证书校验、混合内容
“HTTPS 的安全性体现在三方面：
1. 加密（Confidentiality）：防止中间人窃听，数据在传输过程中不可读；
2. 完整性（Integrity）：防止数据被篡改或损坏，接收端可以验证数据是否被修改；
3. 身份验证（Authentication）：通过服务器证书验证服务器身份，防止假冒网站和中间人攻击。”
 HTTP/1.1 → HTTP/2 → HTTP/3   升级（一步一步解决问题的协议）
不同的 http 协议还限制了我们的资源数量（打包策略）
解决队头阻塞，导致的请求发送不出去
tcp 队头阻塞（网络不好），导致的请求发送过去了（已经到了浏览器，操作系统），但是无法交给应用层（因为 tcp 协议），底层原因就是 tcp 协议是流式传递（ 2GB 会被拆成 上百万个 TCP 段），整个数据（一个 http 请求）完整抵达了才算 ok，才能交给应用层
HTTP/1.1（一句话）  持久连接已经实现了（tcp 连接复用）
因为连接数有限导致的队头阻塞，前端被迫合并资源，牺牲可维护性换性能。
队头阻塞指的是，由于某个请求卡住了，影响后边队伍的请求，根本原因是连接数有限
📌 所以才有：
● JS / CSS 合并
● 雪碧图
● 内联脚本

通过多路复用解决了并发瓶颈，前端可以放心做代码分割，进而实现更佳的按需加载。但是还有 tcp 队头阻塞的问题

二、浏览器缓存机制 （http 缓存和 dns 缓存）
● 缓存分类
  ○ 强缓存：Cache-Control / Expires  有真正的数据和过期时间
  ○ 协商缓存：ETag / Last-Modified  仅仅是一个标志，
● 命中 强缓存 → 直接返回结果，不发请求   静态资源，长期有效
● 强缓存过期 → 进入协商缓存（没修改 304，使用强缓存，修改了 200 ，返回修改数据）
强缓存优先（不发请求），失效后（设置的有效时间， 提醒检查资源更新）才走协商缓存（发请求校验）。  200 更新强缓存，304 继续使用强缓存（重新不过期）
● 浏览器缓存优先级和生命周期
Q1：浏览器缓存有哪些？优先级？
“浏览器缓存主要有内存缓存、Service Worker 缓存、HTTP 缓存（磁盘），优先级从高到低依次是：内存 > SW > HTTP > 网络请求。”
Q2：浏览器缓存的生命周期？
“不同缓存生命周期不同：内存缓存随页面关闭失效，HTTP 缓存受 Cache-Control、ETag / Last-Modified 控制，Service Worker 缓存可自定义，直到被更新或删除。”
● 前端优化点
  ○ 静态资源 hash + CDN
  ○ 减少 304 检查，合理 Cache-Control

三、DNS、TCP/IP、网络传输
● DNS 查询流程
  ○ 本地缓存 → HOSTS → 本地 DNS → 递归查询
dns 是将域名转换为 ip 地址的系统，这个转换过程是耗时的，可以预先 dns 解析，dns-prefetch	
● TCP / TCP 三次握手 & 四次挥手   双工通信（需要关闭两个路线）     几次指的是发送请求的次数，状态。。。
1. syn 是开始请求，fin 是关闭请求，ack 是确认请求
2. 三次握手： 发送请求 syn- 发送应答 ack+syn-发送确认接收应答 ack   状态变为 
3. 四次挥手：客户端发 fin，服务端发 ack ，服务端（处理完成后）发 fin，客户端发 ack
每一方发送 syn 和 fin 都是请求开启自身到对方的连接，得到 ack 后开启连接，
关键区别，服务端接收到请求返回的时候，建立连接的时候 ack和syn是一起的   断开的时候是fin ack分开的，为什么呢：
建立连接的时候没有数据，可以同时开启连接
断开连接的时候可能有残留数据，也就是前后端得自己决定 fin 的时机，中间的 ack 是对前端-》后端 连接断开的确认，这个时候可能后端还在发数据，无法发送 fin，所以 fin 单独发送
● HTTP Keep-Alive / 连接复用
就是持久连接，tcp 复用（减少三次握手，四次挥手 的耗时）
HTTP/1.0
● 默认关闭连接
● 需要加请求头：Connection: keep-alive
HTTP/1.1
● 默认开启持久连接
● 可以通过响应头 Connection: close 显式关闭
HTTP/2
● 默认多路复用 + 持久连接
● 不需要单独设置 Keep-Alive，底层 TCP 连接复用
● 网络延迟来源
  ○ DNS 查询 / TCP 建立 / TLS 握手 / 请求响应 / 渲染阻塞
网络延迟即网络请求的耗时
dns dns-prefetch
tcp 建立：   keep alive tcp 复用
阻塞渲染则是：设置

四、浏览器渲染与请求调度  --通过调度 渲染和请求，优化体验（避免阻塞渲染）
资源的来源（打包做好代码分割后的 chunk，） ，控制首屏懒加载（只加载必须的资源），其余非首屏的常用大量资源可以设置预加载（抖音视频），   对于不常用的非首屏资源只设置动态 import
关键：避免阻塞当前帧的渲染（以及交互）
对于网络层面的优化 其实就是控制资源的加载时机
● 首屏资源优先加载：只请求渲染首屏所需的 JS/CSS/图片，避免一次性加载所有资源导致阻塞。
● 懒加载 / 延迟加载：对非首屏或非关键资源采用懒加载，当用户滚动或访问到对应区域才加载。
● 预加载 / 空闲加载：使用 <link rel="preload"> 或 requestIdleCallback 在浏览器空闲时提前请求非关键资源，提高后续页面交互响应速度。
前提：对于资源做了合理的分割 split
● 前端优化点
  ○ 合并请求 / 分域名 / CDN / HTTP2 / 懒加载

五、前端请求方式
XHR / fetch / axios / 原生 WebSocket （对比差异，从而更好的使用）
xhr 对比后两者，没有基于 promise 实现
xhr 对比 fetch，有请求取消和请求进度的原生能力
fetch  是原生的 api  是封装版本的 xhr，实现了 promise， 但是没有请求取消和请求进度的原生能力
axios 是HTTP 客户端库，需要实例化，对 XMLHttpRequest / fetch 的工程级封装，封装提供了强大的能力：
1. 请求/响应拦截器（统一添加请求头、处理某种错误等） 
2. 全局/实例级默认配置（例如 baseurl，超时设置） 
3. 响应数据自动解析（无需手动json()）
4.便携的取消请求和监控请求的能力   (通过 abortController 取消请求 创建实例，关联 signal  调用实例的 abort 方法)

● 传统 HTTP：客户端发请求 → 服务器响应 → 连接关闭（或 keep-alive 但仍是“请求-响应”模式）。想实现“实时”只能靠轮询（定时问服务器有没有新数据）或长轮询，效率低、延迟高。
● WebSocket：一次HTTP 握手后“升级”成 WebSocket 连接，之后连接一直保持打开，服务器和客户端随时可以互相推送消息，不需要反复建立连接。
● Fetch 特性
  ○ Promise API
  ○ CORS 支持
  ○ 流式处理（ReadableStream）
● 请求优化
  ○ 并发控制 / 防抖 / 节流
  ○ 请求缓存（GET 请求可缓存）

六、跨域
跨域是指浏览器的同源策略限制，当前域名的 JavaScript 代码试图访问其他域名下的资源时会受到限制。
● 同源策略
同源的定义：
协议相同（http/https）
域名相同
端口相同
● 跨域解决方案
  ○ CORS（服务器配置 Access-Control-Allow-Origin）
  ○ postmessage   多窗口间通信
  ○ websocket
